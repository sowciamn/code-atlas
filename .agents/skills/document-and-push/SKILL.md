# 日本語JavaDoc・コミット・プッシュ

CodeAtlasの差分を理解して日本語JavaDocを整備し、検証済みの変更だけを現在のGitブランチへコミットして通常のpushで公開する。

## 1. 作業範囲を確認する

1. `git status -sb`、`git diff`、`git diff --cached`を確認する。
2. 現在のブランチ、`origin`、追跡ブランチを確認する。
3. 差分をファイル単位で読み、ユーザーの依頼に含まれる変更を特定する。
4. 無関係な変更が混在し、対象を安全に判別できない場合はコミット前にユーザーへ確認する。
5. 未追跡の秘密情報、認証情報、ローカル設定、ビルド成果物をコミット対象に含めない。

## 2. 日本語JavaDocを整備する

1. 変更されたJavaコードと関連する既存コードを読み、責務と処理境界を理解する。
2. 公開クラス、公開インターフェース、公開record、公開メソッドへ日本語JavaDocを付ける。
3. recordの各コンポーネントを`@param`で説明する。
4. 引数、戻り値、検査例外、重要な実行時例外を必要に応じて`@param`、`@return`、`@throws`で説明する。
5. 非公開処理には、対応範囲、フォールバック、解析継続条件などがコードだけでは分かりにくい場合だけJavaDocを付ける。
6. 自明な代入、ループ、単純な変換を日本語へ言い換えただけのコメントは追加しない。
7. コメントで未確認の意図を推測しない。実装から確認できる事実だけを書く。
8. 既存JavaDocが実装と一致しない場合は同時に修正する。

## 3. 変更を検証する

1. JavaコードまたはPOMを変更した場合は`./mvnw clean test`を実行する。
2. 実行可能jarやパッケージ構成に影響する場合は`./mvnw clean package`も実行する。
3. 解析結果やMarkdown出力が変わる場合は、次を実行して`docs`を更新する。

   ```bash
   java -jar code-atlas-cli/target/code-atlas-cli.jar generate \
     --source ./examples/springboot-sample \
     --output ./docs
   ```

4. `git diff --check`を実行する。公式Maven WrapperのCRLFなど既知の理由がある警告は、内容を確認して報告する。
5. テストが失敗した場合は原因を修正して再実行する。解消できない場合はコミットとpushを停止する。

## 4. コミットする

1. 最終差分と`git status --short`を再確認する。
2. 依頼対象のファイルだけを明示的に`git add`する。作業ツリー全体が同一目的だと確認できた場合を除き、`git add -A`や`git add .`を使わない。
3. ステージ済み差分を確認し、テスト結果と一致することを確認する。
4. 変更目的を表す簡潔な日本語コミットメッセージを作成する。
5. コミット後にコミットSHAと作業ツリーの状態を確認する。

## 5. プッシュする

1. `origin`と追跡ブランチが設定されていることを確認する。
2. リモートブランチの先行を確認する。先行している場合はfetchして内容を調べ、ユーザーの履歴を失わない方法で統合する。
3. 現在のブランチを`git push`、未追跡なら`git push -u origin <branch>`で公開する。
4. `--force`と`--force-with-lease`は使用しない。
5. push後にリモート参照が新しいコミットを指すことを確認する。

## 停止条件

- push先リポジトリまたは対象ブランチを特定できない。
- Git認証が失敗している。
- 未解決のマージ競合がある。
- テストが失敗している。
- 無関係な変更と依頼対象を安全に分離できない。
- リモート履歴を上書きしなければ公開できない。

停止した場合は、完了済みの作業、コミットの有無、必要なユーザー操作を簡潔に報告する。

## 完了報告

次を報告する。

- 追加または更新したJavaDocの範囲
- 実行した検証と結果
- ブランチ名
- コミットSHAとメッセージ
- push先と反映結果
